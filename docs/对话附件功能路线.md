# 对话附件功能路线

## 第一版目标

第一版聚焦“能干净地把文本附件交给模型理解”，不把文件正文塞进输入框。

- 输入框只显示附件卡片，包含文件名、大小、解析状态和移除按钮。
- 用户消息保存原始可见文本，附件独立保存到 `chat_attachments` 表。
- 文本类文件读取正文后，作为隐藏上下文拼入对应用户消息的 LLM 请求内容。
- 非文本类文件先记录文件名、类型和大小，并在模型上下文里说明暂未提取正文。
- 历史消息展示附件卡片，方便用户知道当时发送了哪些文件。
- 第一版限制：单文件最大 2MB，文本正文最多注入前 40000 字符。

## 当前第一版边界

- 支持文本类文件：txt、md、json、csv、log、常见代码文件、yaml、sql 等。
- 支持 Word `.docx` 正文读取：提取 `word/document.xml` 中的段落文本，作为隐藏上下文发送给模型。
- 暂不解析 PDF、旧版 `.doc`、Excel、图片、音视频、压缩包。
- 暂不做附件预览、下载、重新解析或删除历史附件。
- 暂不按不同模型 Provider 走原生文件上传 API。
- 暂不对附件内容做分块、向量检索或长期知识库沉淀。

## 后续完善清单

### 1. 文件解析能力

- PDF：提取正文、页码和基础结构，保留“第几页”的引用信息。
- Word：在 `.docx` 正文读取基础上，继续补齐标题层级、表格结构、脚注、批注和修订记录。
- Excel/CSV：识别工作表、表头、行数，支持按需截取或汇总。
- 图片：接入支持视觉输入的模型，或先做 OCR。
- 压缩包：只展示文件清单，避免自动展开未知内容带来安全风险。

### 2. 大文件处理

- 超过上下文限制的文件进入分块流程，而不是直接注入 Prompt。
- 增加附件 token 估算，发送前提示预计上下文消耗。
- 对长文件提供“摘要后注入”和“检索后注入”两种策略。
- 与知识库打通：大文件可一键转存为知识条目，再由 RAG 检索。

### 3. Provider 原生文件能力

- 为模型 Provider 增加能力标记：是否支持原生文件输入、图片输入、文件搜索。
- OpenAI 兼容路径：支持 file input 或 file search 时优先使用原生能力。
- 不支持原生文件的 Provider 继续走本地解析后的文本上下文兼容模式。
- 在消息或调试日志中记录本次附件使用的是“原生文件”还是“文本上下文”。

### 4. 附件生命周期

- 历史附件支持查看详情、复制文件名、重新解析。
- 支持删除某条历史消息的附件，并同步影响后续上下文。
- 支持附件内容加密存储或只保存派生文本，不保留原文件。
- 增加附件清理策略，避免本地数据库无限膨胀。

### 5. 用户体验

- 附件卡片支持解析中、解析失败、已截断等状态。
- 拖拽文件到输入框上传。
- 多文件上传与批量移除。
- 发送前提示“这些附件会作为上下文发送给模型”。
- 对不支持解析的文件给出下一步建议，例如“放入知识库”或“换成文本格式”。

### 6. 安全与合规

- 限制可读取的文件大小和类型，避免误传敏感大文件。
- 对附件内容进入模型前提供确认或敏感信息提示。
- 后续如果支持原生上传，需要明确提示文件会发送给第三方模型服务商。
- 审计日志记录附件发送、解析失败、删除等关键动作。
