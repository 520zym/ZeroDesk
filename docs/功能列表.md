# ZeroDesk 功能需求清单 (PRD)

## 设计哲学

ZeroDesk 的 Agent 不是"虚拟员工"，而是**带有特定上下文的 LLM 调用配置**。
用户应该追求用**最少的步骤、最合适的模型、最严格的质量门禁**完成任务，而非为了模拟人类分工而堆叠 Agent。

Multi-Agent 的真正价值不在于模仿人类组织架构，而在于：
- **上下文隔离**：每步只关注相关信息，避免超长上下文的质量退化
- **生成与审核分离**：独立的验收调用比"自己写自己审"质量更高
- **模型路由**：不同步骤匹配不同成本/能力的模型
- **可控性**：多步执行可以在任意节点暂停、检查、纠偏

## 模块一：核心引擎与全局设置 (Core Settings)
**目标**：打通大模型底座，确保工具不被单一平台卡脖子。
* **多源 API 配置中心**：
  * 支持配置多组 Base URL 和 API Key（兼容 OpenAI 接口格式）。
  * 原生支持接入本地模型服务（如 Ollama），方便零成本调试。
* **全局参数预设**：
  * 默认系统级 Temperature、Max Tokens 控制，防止大模型意外"跑飞"消耗大量费用。
* **通用对话入口**：
  * 提供独立的「对话」一级入口，用于日常轻量问答、写作、代码解释和临时思路整理。
  * 对话不强制创建任务，不进入多 Agent 规划流程；用户可直接选择模型并开始聊天。
  * 支持会话列表、历史消息保存、模型切换、温度、最大输出 Tokens 和上下文开关。
  * 与任务中心保持边界：对话解决即时交流，任务中心解决需要计划、分工、执行和验收的复杂工作。

## 模块二：Agent 配置与管理 (Agent Management)
**目标**：定义任务执行步骤的能力边界，每个 Agent 是一个带有特定 Prompt 和工具权限的 LLM 调用配置。
* **手动创建**：
  * 自定义 Agent 基础信息：名称、头像（纯色圆角文字头像）、系统提示词（System Prompt）。
  * 工具权限分配：勾选该 Agent 可调用的工具（如：联网搜索、读写本地文件）。
* **AI 智能拆解**：
  * 用户输入宏观目标（例："整理竞品分析并输出可执行方案"）。
  * 系统自动生成**最小可行的**执行步骤（默认推荐 2-3 步，而非一股脑拆出 5-6 个 Agent），并为每步写好 System Prompt。
  * 推荐原则：能用一个 Agent 完成的不拆成两个，只在"上下文隔离"或"生成与审核分离"有明确收益时才增加节点。
* **配置模板库**：
  * 支持将调试好的 Agent 配置保存为本地模板，一键复用。

## 模块三：智能模型路由 (Model Routing)
**目标**：为每个 Agent 匹配最合适的模型，兼顾质量与成本。模型选择发生在 Agent 层面，不需要用户单独配置路由规则。
* **AI 自动推荐**：
  * 系统根据 Agent 的 Prompt 特征与任务类型，自动从模型池推荐最合适的模型（参考模型的质量等级、速度、价格）。
  * 例：重逻辑推导推荐 GPT-4o / Claude 3.5 Sonnet，轻量级格式化推荐 Qwen-Turbo。
* **用户可手动覆盖**：
  * 在 Agent 配置页面或执行计划视图中，每个 Agent 的模型选项可手动切换。
  * AI 推荐的模型选项旁带有"推荐"视觉标签，最终以用户手动选择为准。
* **平滑降级容灾 (Fallback)**：
  * 当首选模型 API 超时或不可用时，自动无缝切换至预设的备用模型，保障工作流不中断。

## 模块四：任务发起与执行流 (Task Launch & Execution)
**目标**：用户只需要说"干什么"和"花多少钱"，AI 自主决定怎么干——但用户可以审核和调整 AI 的方案。
* **极简任务发起**：
  * 用户输入任务目标 + 选择成本档位（经济/标准/高质量/不限），一键提交。
  * AI 自动生成执行计划（Plan）：规划执行步骤，**优先从用户的 Agent 库中匹配合适的 Agent**，仅在库中无合适人选时才建议新建。
* **可编辑的执行计划（Plan）**：
  * Plan 分为两个独立维度：**执行步骤**（做什么、什么顺序）和 **Agent 团队**（谁来做、用什么能力）。
  * 步骤可增删、排序、编辑描述、重新分配 Agent；Agent 可编辑 Prompt/模型/工具权限。
  * AI 用自然语言解释规划理由，并标注每个 Agent 的来源（复用/新建）。
  * 不使用命名的"协作模式"——Plan 本身就是执行方案，不需要归类。
  * 用户确认 Plan 后才开始执行。
* **Agent 资产沉淀**：
  * 新建的 Agent 在任务完成后可自动沉淀到 Agent 库，变为可复用资产。
  * 整个 Agent 组合可保存为"团队模板"，下次类似任务直接复用。
* **实时执行可视化**：
  * 用柔和色块和线条实时展示当前执行到哪一步、数据流向哪里。
* **人工接管 (Human-in-the-loop)**：
  * **全局暂停键**：随时冻结整个工作流。
  * **下场干预**：直接修改某步输出，或手动插入纠偏指令后恢复执行。

## 模块五：上下文管理与知识库 (Memory & Knowledge Base)
**目标**：解决大模型"金鱼记忆"问题，用分级上下文策略控制 Token 消耗。
* **公共知识库**：
  * 存放敲定版的共识文档（如项目规范、需求边界、接口约定）。
  * 结合轻量级本地向量库，执行时自动检索并提取相关片段，按需注入 Prompt，避免无效 Token 消耗。
* **步骤级记忆区**：
  * 每个 Agent 拥有独立的上下文空间，存放该步骤的推导过程，不污染其他步骤的上下文。
* **版本控制与回退**：
  * 所有文档支持极简时间轴视图。
  * 清晰记录"谁在什么时间修改了什么"，支持一键回退到历史版本，防止 AI 幻觉覆盖核心设定。

## 模块六：成本与效率看板 (Analytics Dashboard)
**目标**：透明化任务执行的成本与效率，找出低效步骤并优化。
* **成本追踪 (Token 消耗)**：
  * 实时总耗费监控面板（Token 消耗量及预估账单金额）。
  * 精确到单个步骤（Agent）的 Token 消耗排行。
  * 根据 Agent 实际使用模型分别套用输入、输出、缓存创建、缓存命中参考价格，汇总任务与历史费用。
  * 支持美元/人民币费用展示切换，人民币金额使用设置中的 USD/CNY 汇率换算。
* **效率指标**：
  * 实时展示各步骤的"平均响应耗时"。
  * 监控"输入输出比"（Input/Output Token Ratio），识别是否输入了大量上下文却只产出少量内容（提示用户优化上下文注入策略）。
* **任务大盘**：
  * 常驻待办池（Task Queue），显示当前工作流的整体进度条。

## 模块七：界面与交互体验 (UI/UX)
**目标**：提供一个能让人长时间专注、不产生视觉疲劳的舒适工作台。
* **极简扁平风**：告别企业级软件的厚重感，界面大面积留白。
* **莫兰迪配色体系**：
  * 背景：暖灰白。
  * 主色调：低饱和度的灰绿、雾霾蓝等。
  * 字体：深灰而非纯黑。
* **沉浸式控制台**：
  * 左侧导航，中间执行流/消息流，右侧指标面板。复杂的参数配置全部收纳进二级设置页。
  * 左侧导航支持收缩为图标栏，在控制台场景中降低存在感并释放横向空间。

## 模块八：任务质量门禁与验收 (Quality Gate & Validation)
**目标**：避免"看起来很努力，结果全是废话"，让工作流具备可验证质量。
* **节点级验收器 (Per-step Validator)**：
  * 每个关键步骤输出后，先经过验收规则检查，再决定是否流转到下游。
  * 支持基础规则：最小字数、关键词覆盖、禁用词、格式完整性。
* **结构化输出约束 (Schema Output)**：
  * 支持为任务节点绑定 JSON Schema / 固定字段模板（如：结论、证据、风险、下一步）。
  * 当输出不符合结构约束时，自动触发"返工提示词"重试。
* **人工检查点 (Human Checkpoint)**：
  * 对关键节点设置"必须人工确认"开关，确认后才允许继续执行。
  * 支持一键"驳回并附修改意见"，将反馈自动注入下一轮上下文。

## 模块九：稳定性与容灾治理 (Reliability & Resilience)
**目标**：让系统在真实网络与模型波动环境下稳定可用，不因一次故障全盘中断。
* **统一重试策略**：
  * 针对超时、限流、网络抖动提供可配置重试次数与指数退避。
  * 区分"可重试错误"与"不可重试错误"，避免无意义死循环。
* **多级降级与接管**：
  * 首选模型失败后自动切换备用模型；连续失败后自动降级为低成本安全模式。
  * 达到失败阈值时，弹出人工接管入口（暂停、改指令、换模型、继续跑）。
* **会话级恢复 (Session Resume)**：
  * 应用异常退出后，支持恢复上次任务上下文、执行节点状态与关键日志。
  * 提供"从失败节点续跑"而非整条链路重跑，节省时间和 Token。

## 模块十：Prompt 工程与模板工坊 (PromptOps & Templates)
**目标**：将 Prompt 调优从"玄学"变成可追踪、可复用、可回滚的工程能力。
* **Prompt 版本管理**：
  * 每个 Agent 的 System Prompt 支持版本历史、差异对比、备注标签与一键回滚。
  * 记录"版本变更 -> 质量表现/成本变化"的关联，支持经验沉淀。
* **任务模板库 (Workflow Recipes)**：
  * 新增任务流程模板（如：需求拆解、技术调研、代码评审、运营复盘）。
  * 模板支持参数化（输入主题、风格、预算），一键复用并自动生成任务骨架。
* **可解释路由日志**：
  * 记录每次模型选择与降级的理由（成本、延迟、质量偏好、历史表现）。
  * 让用户可追溯"为什么这次选它"，减少黑盒感。

## 模块十一：安全边界与审计合规 (Security & Audit)
**目标**：在本地优先的前提下，建立可控、可追责、可安心长期使用的系统边界。
* **密钥与凭据安全**：
  * API Key 本地加密存储，支持系统级凭据管理与环境变量覆盖。
  * 提供"密钥健康检查"（可用性、权限、余额提示）与脱敏展示。
* **工具权限沙箱**：
  * 对联网、文件读写、命令执行设置最小权限原则与显式授权。
  * 支持按 Agent 维度配置白名单目录/域名，防止越权访问。
* **操作审计日志**：
  * 记录关键行为：谁在何时修改了 Agent、Prompt、模型绑定与工作流配置。
  * 支持按任务维度导出审计记录，便于复盘与问题追踪。

## 模块十二：Skills 系统与市场 (Skills Hub & Marketplace)
**目标**：让 Agent 的工具能力可插拔、可复用、可扩展，形成长期增长的生态。
* **Skills 作用域分层**：
  * **Agent 私有 Skills**：仅当前 Agent 可调用，适合步骤专属能力（如"质量审查器"）。
  * **任务组共享 Skills**：同一任务组内所有 Agent 可调用，适合通用能力（如"数据清洗器"）。
  * **全局已安装 Skills**：在本地全局可见，可按需分配到任意任务组或 Agent。
* **可视化装配与继承关系**：
  * 在 Agent 面板可视化添加/移除私有 Skills，并展示来源标签（私有/继承自任务组/全局分配）。
  * 在任务组面板维护共享 Skills 池，支持批量授权。
* **冲突解析与优先级**：
  * 当出现同名 Skill 或同类能力冲突时，按"Agent 私有 > 任务组共享 > 全局已安装"优先级解析。
  * 冲突处理结果在执行日志中可追溯，便于排障。
* **内置 Skills 市场入口**：
  * 在工具内提供市场页：搜索、安装、升级、卸载、启用/禁用 Skills。
  * 支持查看 Skill 的简介、版本、兼容性、权限声明与更新日志。
* **安全安装与运行沙箱**：
  * 安装前展示权限申请（文件、网络、命令等），需显式确认后才可启用。
  * 支持 Skill 版本锁定、签名校验（或可信来源校验）与一键回滚。
  * 运行时权限受沙箱约束，防止越权访问本地敏感资源。

## 模块十三：发布管理与安装指引 (Release Management)
**目标**：让每次版本发布不只产出安装包，也能清楚告诉用户改了什么、怎么安全安装和升级。
* **详细变更说明**：
  * 每个 GitHub Release 都应包含面向用户的详细变更，而不是只依赖 tag message 或 commit message。
  * 变更内容按类型组织，例如新增功能、修复问题、体验优化、兼容性变化和已知问题。
  * 对影响数据、配置、工作流或使用习惯的变更，需要明确说明影响范围与建议操作。
* **安装包使用注意事项**：
  * Release 页面需要列出不同平台安装包的适用场景，例如 Apple Silicon Mac、Intel Mac、Windows x64。
  * 对未签名或未公证的 macOS 安装包，需要说明首次打开时可能遇到的系统拦截和处理方式。
  * 说明升级安装、数据目录、备份建议、回退方式和已知兼容性限制。
* **发布说明模板化**：
  * 建立固定 Release Notes 模板，确保每次发版都包含版本摘要、详细变更、安装说明、升级注意事项和下载指引。
  * 后续可接入自动生成草稿，但发布前必须人工检查用户可读性与安装风险提示。
